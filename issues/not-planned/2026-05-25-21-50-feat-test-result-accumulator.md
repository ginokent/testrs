# テスト結果アキュムレータを追加する

- Priority: Low
- Created: 2026-05-25 20:59 JST
- Completed:
- Model: -
- Branch: feature/add-test-result-accumulator

## 目的

複数プロパティを走らせて、最初の失敗で panic させずに全失敗を
まとめてレポートできるようにする。

## 経緯

`BACKLOG.md` の 🟡 バックログ項目「テスト足回り」から
`issues/` 配下へ移行する。

## 優先度根拠

BACKLOG.md 上で「典型ワークフローでの影響は小」とされている。
また「大規模な refactor が必要」との注記もあるため Low とする。

## 現状・問題

現状、最初に失敗したプロパティで panic がスローされ、後続の
プロパティ結果は得られない。複数の失敗を 1 ランで把握できない。

## 完了条件

- 複数の `forall` / `run` を集約する `Accumulator` 相当の API を
  追加する
- 集約結果から全失敗を列挙できる
- 集約呼び出しのオプトイン化 (既存挙動はデフォルトで維持) と、
  集約挙動の単体テストを追加する

## 解決方法

ランナーが panic を内部で `Outcome::Failed` に変換するパスは
既に存在するので、これを集約 API でループする形にする。
大規模 refactor が必要との注記あり。

## 設計方針

- no-deps 方針を維持する
- 既存の `run` / `forall` の挙動 (失敗で panic) は変えず、
  オプトインの集約 API として追加する
- 集約された失敗のレポーティングは既存の `Outcome` フォーマットを
  再利用する

## クローズ (not-planned, 2026-06-05)

費用対効果が見合わないため not-planned とする。

- issue 自身が「**大規模 refactor が必要**」かつ「典型ワークフローでの
  影響は小」と明記しており、オーバーエンジニアリングの典型
- 「最初の失敗で panic させず全失敗をまとめる」という目的は、各
  `#[propcheck]` プロパティが独立した `#[test]` として展開されるため、
  cargo のテストハーネスが既にプロパティ単位で全件実行し全失敗を個別
  レポートすることで実質的に満たされている
- 集約値オブジェクトを別途持つ必要が出るのは実需要が示されてから判断
  すれば足りる
